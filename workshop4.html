01

The big shift

You've spent three workshops learning to write the code yourself — bash, git, Python. This one is different. An AI assistant can now write most of that code faster than you can type it. So the skill worth an hour of your time isn't how to prompt. It's how to stay in charge of work you didn't type.

Picking up from Workshop 3. You have a shell, a repo, Python, and a result you trust because you derived it — 1,343 genes up in bone marrow. Keep that number close: it's how you'll keep an AI honest today.

Where the bottleneck went

Before
  • Writing code was the slow part
  • You thought as you typed
  • Understanding grew while you implemented
  • Time went into debugging
Now
  • Implementation is nearly instant
  • Thinking and planning move up front
  • Reviewing and evaluating is the job
  • The bottleneck is judgment, not keystrokes

Your new job description

The assistant is the fast typist. These five roles stay yours — and they're the ones that decide whether the science is right.

Architectdefine the structure before a line is written
Reviewerread and evaluate every line it produces
Plannerbreak the work into small, checkable steps
Editorrefine, reject, and demand rewrites without mercy
Testerverify behavior — not just that it runs

When not to reach for an assistant

Knowing this is a mark of skill, not weakness. Skip the AI when:

  • you don't understand the problem well enough to judge the output — you'd be trusting blind;
  • the domain needs expertise you can't verify (a stats method you've never checked by hand);
  • a subtle bug is expensive — and in a result headed for a paper, it is;
  • reviewing the AI's version would take longer than just writing the ten lines yourself.
The through-line of this whole workshop: an assistant is only safe when you can check what it gives you. Everything below is about never losing that ability.
02

How the tools actually work

Ignore the brand names for a moment. A handful of ideas are true of every assistant — ChatGPT, Claude, Gemini, Copilot — and knowing them is most of what separates people who get good code from people who get confident nonsense.

The four ideas that transfer everywhere

Context windowthe assistant's working memory — think RAM. Everything you've said this session sits in it, and quality degrades as it fills. Long, wandering chats get worse, not better. Start a fresh chat for a new task.
Plan before you promptthe biggest wins come from thinking first. Describe the goal, the inputs, and the check — then ask for code. Let it ask you questions; it'll surface edge cases you missed.
One task per promptask for five things and you get five half-right things. Ask for one, verify it, move on. Small verified steps beat one big leap.
Be concretename your files and columns (log2FoldChange, padj), give the exact threshold, show an example of the output you want. Ambiguity is what it fills with hallucination.

Pick your assistant — all free

This workshop is deliberately tool-agnostic. The habits transfer; the logos don't matter. Any of these works for the lab:

ChatGPT · chat.openai.com Claude · claude.ai Google Gemini · gemini.google.com Gemini in Colab · the ✦ button, in the lab itself GitHub Copilot · free on the Student Pack
Two are worth calling out for you specifically. Gemini is built into Colab — free for every user, right where today's lab runs, so there's nothing to install. And GitHub Copilot is free for students: you already made a GitHub account in Workshop 2, so verify it through the Student Developer Pack and it lives in your editor from here on.
Going deeper — the power-user vocabulary (optional)

Once the basics are second nature, a whole layer opens up: Plan Mode (research-then-build), project instruction files, hooks that auto-run your tests, reusable skills, subagents and swarms for parallel work, and MCP — the protocol that lets an assistant read your files, query a database, or drive a browser. Powerful, and genuinely overkill for renaming a variable. You'll grow into them when a task actually needs them.

The framing in this workshop follows Dan Larremore's Principles for Writing Code with LLMs — worth reading in full when you want that deeper layer.

03

The verify-first loop

This is the one non-negotiable idea in the whole session. AI code that runs is not AI code that's right. You close that gap by checking every result against something you already trust — a known number, a second file, a test you wrote. Here's the loop, then you'll catch a real bug with it.

1 · Askone concrete task, with the columns and thresholds named
→
2 · Readevery line — does it do what you asked, or something near it?
→
3 · Verifyrun it against a number or file you already trust
→
4 · Accept / rewritematches? keep it. Off? reject or demand a rewrite

The trap is step 3. It's tempting to skip it when the code looks fine. Skipping it is exactly how a wrong number ends up in a figure.

Try it. Below is a function an assistant wrote to "count the up-regulated genes" in a little sample table. It runs. It looks reasonable. You happen to know 3 of these genes are truly up-regulated. Run it — then fix it.

Start the Python engine — downloads ~15 MB once (a few seconds), then the cell runs instantly and offline. Kick it off now; it loads while George talks.

SPOT THE BUGDoes this really count "up"? 

    
stuck? the tell
      
It prints 5, not 3. The bug is abs(fc) > 1: "up" means fc > 1, but
         abs also sweeps in the two strongly down genes. Change abs(fc) > 1 to
         fc > 1 and run again — now it's 3, and it matches what you knew.

  


  
This is not a toy. That exact abs() mistake, run on your real bone-marrow
    table, reports 2,189 up-regulated genes instead of 1,343 — 846 phantom genes, because it counted
    everything going down as if it were going up. And there's a subtler cousin waiting in the Colab lab: swap
    padj for pvalue and the count slips to 1,445 — close enough that you'd never blink,
    wrong enough to matter. You only ever catch these by checking against a number you own.


  

    
The rest of the discipline

    

      
        
Let it debugpaste the error and say "fix," nothing else. Your guess about the cause
          often sends it the wrong way. If it fails twice, start fresh rather than argue in circles.

        
Demand rewriteswhen something works but is ugly, don't patch it: "knowing what you know
          now, scrap this and write the clean version." The rewrite beats the patch almost every time.

        
Tests are the netwrite the check first from an answer you trust, then let the AI
          write code to pass it. You'll meet this head-on in the Expert lab.

        
Diff before you trustgit diff is your review tool. Never let an assistant
          make sweeping changes without a commit to fall back to — you already know how (Workshop 2).

        
Watch the importsassistants invent packages that don't exist and mishandle keys and
          auth. Treat every dependency it adds like a stranger's code, because it is.

      
    

    
One line to carry out of here: treat AI output like a sharp junior colleague's pull request —
      trust the speed, verify the substance.

    05
The capstone — push it, honestly

    
  

  
One rule closes the bootcamp: when AI helped you make a result, you say so — plainly, in the
     repo, so the next person knows what the machine did and what you checked. It costs one file, and it's simply how
     honest computational work is done now.


  

    
Add the AI-assisted analysis to your repo

    
The Expert lab has you write an AI_NOTES.md — what you
       asked for, how you verified it, what you changed by hand. Download your notebook (File → Download → .ipynb),
       move both to Fiji beside your zfp36l2-analysis repo, and run the loop you already know:

    

      
      
$ cd ~/zfp36l2-analysis
$ git add analysis.ipynb AI_NOTES.md
$ git commit -m "Add AI-assisted tissue-count analysis, verified against published landmarks"
$ git push

    

    
That single commit is the whole bootcamp in one object: real data, under version control, analyzed
      in Python, accelerated by AI, and documented honestly. bash → git → Python → AI, closed.

  


  

    
Where you started, where you are

       move a real dataset around a filesystem, put it under version control, analyze it in Python, and point an AI
       assistant at the whole thing without ever losing the ability to tell whether it's right. That last part —
       verifying against ground truth you own — is the skill that will still matter when today's tools are long
       replaced. You already have it.
