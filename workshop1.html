02

The command line

Linux powers nearly every server, cloud, and supercomputer on Earth — Fiji included. The command line is how you talk to it: faster than clicking for big data, scriptable, reproducible, and controllable from anywhere over SSH. Nobody memorizes all of it — getting good at looking things up is the skill.

Anatomy of a command

$ ls -l /home/
  │  │    └─ argument — what to act on
  │  └─ option/flag — how to act (here: long format)
  └─ command — what to do
Almost every command answers --help (or -h), and man command opens the full manual — q to quit it.

Core commands — the searchable cheatsheet

manmanual for any command

Combining commands — where it gets powerful

$ ls -l | grep ".txt"   # | pipes output of one command into the next
$ ls -l > listing.txt    # > sends output to a file (>> appends)
$ cd data && ls -lh      # && chains commands: second runs if first succeeds
This shell is bash — a full scripting language with variables, loops, and functions. Workshop exercises build up to writing your first script.

Editing files in the terminal

$ nano notes.txt   # simple: Ctrl+O save · Ctrl+X exit

vim exists too — powerful, steeper curve. Trapped inside it? Esc then :q! — everyone's been there.

03

Fiji — your supercomputer

Fiji is CU Boulder's BioFrontiers high-performance computing (HPC) cluster — bioinformatics, genomics, ML, simulations. Free for first-year IQ Biology students, managed by BioFrontiers IT (BIT). Jes covered the big picture at 2:00; here's what you need at your fingertips.

Where files live — and why it matters

PathWhat it's for
~/ (home)Permanent, small — code, configs, notes. Not big data.
/scratch/Users/<identikey>Temporary, high-speed I/O — where jobs read/write big data. Not backed up.
/scratch/Shares/Temporary group space (request access)
/Shares/ · /projects/Shared group space (request access)
 
[you@fiji ~]$ du -sh *     # how big is everything here?
[you@fiji ~]$ df -h .      # how full is this filesystem?

The one etiquette rule

Never run heavy computation on the login node. The login node is the shared front door — everyone's on it. Real work goes through Slurm (next section), which hands you a compute node. Break this rule and BIT will email you; everyone gets that email once.

Software — modules & conda

[you@fiji ~]$ module avail          # what's installed?
[you@fiji ~]$ module load python/3.11
[you@fiji ~]$ module list           # what's loaded now?
Custom Python/R environments → conda. If conda misbehaves after a module load, run eval "$(conda shell.bash hook)" first (thanks, Ben Braun).

Moving files on and off Fiji

Run these from your laptop's terminal, not from Fiji:

$ scp identikey@fiji.colorado.edu:~/results.csv .     # download
$ scp data.csv identikey@fiji.colorado.edu:~/         # upload
$ rsync -avhP identikey@fiji.colorado.edu:~/results/ ./results/

rsync > scp for big or repeated transfers — resumable, shows progress, skips unchanged files

04

Slurm — asking for compute

Slurm — Fiji's workload manager (the name began as Simple Linux Utility for Resource Management) — is the traffic controller: it queues jobs, allocates CPUs/memory/time, and balances load across compute nodes. You describe what you need; Slurm finds you a machine.

Three ways to run

TypeCommandWhen
Batchsbatch job.shThe workhorse — submit a script, walk away
Interactivesrun / sallocDebugging, quick tests — srun --pty bash drops you straight onto a compute node; salloc reserves nodes you then attach to
Monitoringsqueue, sacctWatching what's running / what ran

Build your first job script — live

Fill in the fields; the script writes itself. Copy it into nano job.sh on Fiji, then sbatch job.sh.

[you@fiji ~]$ squeue --me            # my running/queued jobs

[you@fiji ~]$ scancel <job_id>       # kill a job

B1. Find out where you are, then list everything in your home directory — including hidden files.

      
hint
One command tells you where you are; ls has a flag for hidden files (they start with .).

      
solution
$ pwd
$ ls -a

    
B2. Build this structure in one command: a bootcamp folder containing data, scripts, and results folders.

      
hint
mkdir -p creates nested paths; braces {a,b,c} expand to multiple names.

      
solution
$ mkdir -p bootcamp/{data,scripts,results}
# or: mkdir bootcamp bootcamp/data bootcamp/scripts bootcamp/results
# (practice sandbox: use the second form — {…} brace expansion is a real-shell feature)

    
B3. Move into bootcamp/data and create an empty file called notes.txt. Prove it exists with a long listing.

      
hint
cd, touch, then ls -l.

      
solution
$ cd bootcamp/data
$ touch notes.txt
$ ls -l

    
B4. Put the line day one of grad school into notes.txt without opening an editor, then print the file.

      
hint
echo plus the > redirect; cat to print.

      
solution
$ echo "day one of grad school" > notes.txt
$ cat notes.txt

    
B5. Make a copy of notes.txt called backup.txt, then rename the copy to archive.txt.

      
hint
Copying and renaming are two different commands — and rename is really "move".

      
solution
$ cp notes.txt backup.txt
$ mv backup.txt archive.txt

    
B6. Delete archive.txt. Then go up one level and delete the whole results folder. Remember: there is no trash can.

      
hint
Files: rm. Directories need a flag. .. means "up one".

      
solution
$ rm archive.txt
$ cd ..
$ rm -r results

    
B7. Use the manual to find the ls flag that sorts by modification time. (On Fiji: man ls, press / to search, q to quit.)

      
hint
Search the man page for the word "modification".

      
solution
$ man ls    # it's -t → try: ls -lt

  


  

    
I1. Generate a file containing the numbers 1–100, one per line, then count its lines to confirm.

      
hint
seq prints number sequences; redirect it. wc counts.

      
solution
$ seq 1 100 > numbers.txt
$ wc -l numbers.txt

    
I2. Print lines 16–20 of numbers.txt — nothing else — using a pipe.

      
hint
First 20 lines, then the last 5 of that.

      
solution
$ head -n 20 numbers.txt | tail -n 5

    
I3. Create genes.txt with these lines: gata4, kcnq1, gata4, myh7, kcnq1, gata4 (one per line). Now produce a count of how many times each gene appears, most frequent first.

      
hint
printf "a\nb\n" writes multiple lines. uniq -c only counts adjacent duplicates — sort first. Then sort the counts numerically, reversed.

      
solution
$ printf "gata4\nkcnq1\ngata4\nmyh7\nkcnq1\ngata4\n" > genes.txt
$ sort genes.txt | uniq -c | sort -rn
      3 gata4
      2 kcnq1
      1 myh7

    
I4. Make a tiny CSV — expr.csv with header gene,count and two data rows — then extract just the gene column.

      
hint
cut with a delimiter flag and a field number.

      
solution
$ printf "gene,count\ngata4,42\nmyh7,7\n" > expr.csv
$ cut -d',' -f1 expr.csv

    
I5. How many lines in genes.txt contain "gata" — regardless of case? One pipeline, one number out.

      
hint
grep has a case-insensitive flag and a count flag — you may not even need the pipe.

      
solution
$ grep -ic "gata" genes.txt   # or: grep -i "gata" genes.txt | wc -l

    
I6. Pack your whole bootcamp folder into a compressed archive, then list the archive's contents without extracting it.

      
hint
Create: -cvzf. List: swap create for -t.

      
solution
$ tar -cvzf bootcamp.tar.gz bootcamp/
$ tar -tzf bootcamp.tar.gz

  


  

    
A1. Write your first script: hello.sh that prints running on <hostname>. Make it executable and run it.

      
hint
Start the file with #!/bin/bash. $(command) embeds a command's output. chmod +x makes it runnable.

      
solution
$ nano hello.sh     # contents:
#!/bin/bash
echo "running on $(hostname)"
$ chmod +x hello.sh && ./hello.sh

    
A2. Write a loop (in a script or one line) that creates sample_1.txt through sample_5.txt, each containing its own number.

      
hint
for i in $(seq 1 5); do … ; done — and $i inside.

      
solution
$ for i in $(seq 1 5); do echo "$i" > sample_$i.txt; done
$ ls sample_*

    
A3. On Fiji: load a Python module and prove it works by printing the version from Python itself.

      
hint
module avail to see names, module load, then python --version.

      
solution
[you@fiji ~]$ module load python/3.11
[you@fiji ~]$ python --version

    
A4. Submit your first real job: use the script builder above to make job.sh that runs sleep 60, submit it, and catch it in the queue before it finishes.

      
hint
Builder → set command to sleep 60 → nano job.sh, paste → sbatch job.sh → squeue --me, fast.

      
solution
[you@fiji ~]$ sbatch job.sh
Submitted batch job 1234567
[you@fiji ~]$ squeue --me

    
A5. When A4's job finishes, find it in the accounting records and check how efficiently it used what you requested.

      
hint
sacct lists finished jobs with IDs; seff takes an ID.

      
solution
[you@fiji ~]$ sacct
[you@fiji ~]$ seff 1234567    # sleep uses ~0% CPU — that's the point

    
A6. Copy numbers.txt to your scratch space, verify it arrived, then bring a copy back named numbers_from_scratch.txt.

      
hint
Your scratch is /scratch/Users/$USER — and $USER saves typing your identikey.

      
solution
[you@fiji ~]$ cp numbers.txt /scratch/Users/$USER/
[you@fiji ~]$ ls /scratch/Users/$USER/
[you@fiji ~]$ cp /scratch/Users/$USER/numbers.txt numbers_from_scratch.txt

  


  

    
E1. Turn A2 + A4 into an array job: one script, five parallel tasks, where task N reads sample_N.txt and appends processed by task N to it.

      
hint
#SBATCH --array=1-5, and inside the script the variable $SLURM_ARRAY_TASK_ID is the task's own number.

      
solution
#!/bin/bash
#SBATCH --array=1-5
#SBATCH --time=00:05:00
#SBATCH --mem=1G
echo "processed by task $SLURM_ARRAY_TASK_ID" >> sample_${SLURM_ARRAY_TASK_ID}.txt
$ sbatch array.sh && squeue --me   # five tasks, one submission

    
E2. From your laptop: pull your five processed samples off Fiji with rsync, edit one locally, and push it back. One flag set, both directions.

      
hint
-avhP: archive, verbose, human, progress+resume. Order of arguments decides direction.

      
solution
$ rsync -avhP identikey@fiji.colorado.edu:~/sample_*.txt ./samples/
$ rsync -avhP ./samples/sample_1.txt identikey@fiji.colorado.edu:~/

    
E3. Stop typing the full login. Create an SSH alias so that ssh fiji just works from your laptop.

      
hint
Edit ~/.ssh/config on your laptop: a Host block with HostName and User.

      
solution
# ~/.ssh/config (on your laptop)
Host fiji
    HostName fiji.colorado.edu
    User identikey
$ ssh fiji

    
E4. One pipeline, no intermediate files: generate 1,000 random numbers between 1 and 50, then print the three most common values with their counts.

      
hint
shuf -i 1-50 -rn 1000 generates; then it's I3's pipeline plus a head.

      
solution
$ shuf -i 1-50 -rn 1000 | sort | uniq -c | sort -rn | head -3

    
E5. (Teaser for later workshops) SSH port-forwarding lets a Jupyter server on Fiji show up in your laptop's browser. Read man ssh on the -L flag and write down — in one sentence — what ssh -L 8888:localhost:8888 fiji does. Ask an instructor to check your sentence.

      
hint
Think: "my laptop's port 8888 becomes a tunnel to ___'s port 8888".

      
solution
"Anything my laptop sends to localhost:8888 is tunneled through SSH to port 8888 on Fiji" — i.e., Fiji's Jupyter appears in your local browser.
