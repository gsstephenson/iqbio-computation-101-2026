01

Why version control

You already know the problem — you've just been solving it badly.

analysis_final.py
analysis_final_v2.py
analysis_final_v2_REAL.py
analysis_final_v2_REAL_use_this_one.py
analysis_final_v2_REAL_use_this_one_FIXED.py

Git replaces that pile with one file and a time machine. Every commit is a snapshot you can return to, compare against, and share. Three problems die at once:

ProblemGit's answer
"I broke it and can't get back"Every commit is a restore point
"Which version did the figures come from?"The commit hash says exactly — reproducibility, for free
"My laptop died" / "we can't work together"GitHub keeps the synced copy everyone (and every machine) shares

Picking up from Workshop 1: you left with a zfp36l2-analysis folder and a findings.md on Fiji. Today you put that project under version control and push it to your own GitHub — the repo your Friday analysis (and Monday's AI work) will land in.

02

Speak git

Same trick as Workshop 1: git isn't magic words, it's a language. Git sentences are two-word verbs — git gets the computer's attention, the next word is the order. Tap any word to hear it in English.

verbadverb (flag)object person & machinegrammar particles
git commit -m "Add first analysis script"

git push origin main

git@github.com:gsstephenson/zfp36l2-analysis.git

The nouns — a pocket dictionary

WordMeans
repoone project + its entire history, living in a hidden .git folder
commitone snapshot, with a message and an ID (the 7-character hash)
stagingthe loading dock — git add puts changes there; commit ships whatever's on the dock
remote / origina copy of the repo elsewhere / the nickname for the one you cloned from
branch / maina parallel timeline of commits / the default timeline
clone · push · pullcopy a repo down · send commits up · fetch commits down
~/.gitconfigwhere your identity (name, email, default branch) lives — global; each repo's own .git/config can override it
Read git sentences aloud and the fear evaporates: git status — “git: how are things?” git log — “git: read me the diary.” And when a verb stumps you, git ships an excellent built-in dictionary: git help commit (or man git-commit) — same man habit as Workshop 1.

SSH keys — the ceremony

The one-time setup that historically costs new students weeks — done here, live, together, in twenty minutes. Full step-by-step with screenshots lives in Guide 2; this is the ceremony itself.

[you@fiji ~]$ ssh-keygen -t ed25519 -C "your_email@colorado.edu"   # Enter, Enter, Enter
[you@fiji ~]$ cat ~/.ssh/id_ed25519.pub   # copy the WHOLE line it prints

GitHub → avatar → Settings → SSH and GPG keys → New SSH key → title it fiji → paste → Add SSH key. Then the moment of truth:

[you@fiji ~]$ ssh -T git@github.com

Hi YOUR-USERNAME! You've successfully authenticated, but GitHub does not provide shell access.

The private key (id_ed25519, no .pub) never leaves Fiji, never gets pasted anywhere, never gets shared. The .pub one is the only one GitHub sees.

04

The daily loop

Ninety percent of your git life is four commands in a circle: status → add → commit → push. Watch a file travel — click the buttons.

working directory
(your files)analysis.py
staging
(the loading dock)
local repo
(committed history)
github
(shared copy)

You edited analysis.py — it's sitting in your working directory, modified. Ask git what it thinks.

The same loop, for real

$ git status                          # “git: how are things?”
$ git add analysis.py                 # onto the loading dock
$ git commit -m "Describe what changed"
$ git push                            # ship it to GitHub
Commit early and often — small commits with honest messages. A commit is free; losing an afternoon of work isn't.

When timelines disagree — merges & conflicts

Branches are parallel timelines, and joining them (git merge) has three outcomes:

SituationWhat git does
Only one timeline movedFast-forward — the pointer just slides ahead; no new commit
Both moved, different files/linesMerge commit — a new commit with two parents ties the timelines together
Both changed the same linesConflict — git stops and asks you to choose

A conflict isn't an error — it's git asking a question only a human can answer. It marks the disputed lines right in the file:

<<<<<<< HEAD
funding for computation
=======
funding for microscopy
>>>>>>> microscopy

your branch's version is above the =======, the incoming branch's below — exactly what A5 will show you

Fix = edit the file to the text you actually want (delete the marker lines), then git add it and git commit. Exercise A5 walks you into one on purpose.

Before you set up GPG — is it worth it?

      
SSH already gets your code to GitHub securely. GPG signing is different: it proves the commit was authored by you and earns it a green Verified badge. It's optional for this bootcamp — here's the trade-off before you spend the setup time.

      

        
Pros
    
          
  • A green Verified badge — cryptographic proof the commit is really from you.
    • 
          
  • Stops impersonation: anyone can put your name/email on a commit, but nobody can forge your signature.
    • 
          
  • Expected in some labs and for published or collaborative research code.
    • 
        

        
Cons
    
          
  • Fiddlier than SSH: key + agent/pinentry, exporting the public key, matching your GitHub email.
    • 
          
  • On a shared cluster you have to guard the private key and passphrase.
    • 
          
  • Not needed for access — SSH handled that. This is about authorship, not permission.
    • 
        

      

      
Bottom line: do it if you'll publish or collaborate on code; skip it otherwise. Full steps in Guide 4.
