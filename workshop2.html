01

Why version control

You already know the problem — you've just been solving it badly.

analysis_final.py
analysis_final_v2.py
analysis_final_v2_REAL.py
analysis_final_v2_REAL_use_this_one.py
analysis_final_v2_REAL_use_this_one_FIXED.py

Git replaces that pile with one file and a time machine. Every commit is a snapshot you can return to, compare against, and share. Three problems die at once:

ProblemGit's answer
"I broke it and can't get back"Every commit is a restore point
"Which version did the figures come from?"The commit hash says exactly — reproducibility, for free
"My laptop died" / "we can't work together"GitHub keeps the synced copy everyone (and every machine) shares

Speak git

Same trick as Workshop 1: git isn't magic words, it's a language. Git sentences are two-word verbs — git gets the computer's attention, the next word is the order. Tap any word to hear it in English.

verbadverb (flag)object person & machinegrammar particles
git commit -m "Add first analysis script"

git push origin main

git@github.com:gsstephenson/my-first-analysis.git

The nouns — a pocket dictionary

WordMeans
repoone project + its entire history, living in a hidden .git folder
commitone snapshot, with a message and an ID (the 7-character hash)
stagingthe loading dock — git add puts changes there; commit ships whatever's on the dock
remote / origina copy of the repo elsewhere / the nickname for the one you cloned from
branch / maina parallel timeline of commits / the default timeline
clone · push · pullcopy a repo down · send commits up · fetch commits down
Read git sentences aloud and the fear evaporates: git status — “git: how are things?” git log — “git: read me the diary.”
03

SSH keys — the ceremony

The one-time setup that historically costs new students weeks — done here, live, together, in twenty minutes. Full step-by-step with screenshots lives in Guide 2; this is the ceremony itself.

[you@fiji ~]$ ssh-keygen -t ed25519 -C "your_email@colorado.edu"   # Enter, Enter, Enter
[you@fiji ~]$ cat ~/.ssh/id_ed25519.pub   # copy the WHOLE line it prints

GitHub → avatar → Settings → SSH and GPG keys → New SSH key → title it fiji → paste → Add SSH key. Then the moment of truth:

[you@fiji ~]$ ssh -T git@github.com

Hi YOUR-USERNAME! You've successfully authenticated, but GitHub does not provide shell access.

The private key (id_ed25519, no .pub) never leaves Fiji, never gets pasted anywhere, never gets shared. The .pub one is the only one GitHub sees.

04

The daily loop

Ninety percent of your git life is four commands in a circle: status → add → commit → push. Watch a file travel — click the buttons.

working directory
(your files)analysis.py
staging
(the loading dock)
local repo
(committed history)
github
(shared copy)

You edited analysis.py — it's sitting in your working directory, modified. Ask git what it thinks.

The same loop, for real

$ git status                          # “git: how are things?”
$ git add analysis.py                 # onto the loading dock
$ git commit -m "Describe what changed"
$ git push                            # ship it to GitHub
Commit early and often — small commits with honest messages. A commit is free; losing an afternoon of work isn't.

06

Exercises

Beginner runs fine in the sandbox above; Intermediate and Advanced want the real thing on Fiji. Hints first, solutions when you need them — check off as you go.

B1. Prove git knows who you are: print your configured name, email, and default branch (set in Workshop 1's Guide 3 — set them now if missing).

hint

git config --get reads one setting at a time.

solution
$ git config --get user.name
$ git config --get user.email
$ git config --get init.defaultBranch

B2. Make a folder git-practice, turn it into a repo, and read what git status says about a brand-new repo — every word.

hint

mkdir, cd, git init, git status.

solution
$ mkdir git-practice && cd git-practice && git init
$ git status

On branch main — No commits yet — nothing to commit

B3. Create README.md with one line of content. Run git status before and after git add — watch the file move from untracked to staged.

hint

echo "text" > README.md, then status → add → status.

solution
$ echo "# Git practice" > README.md
$ git status   # Untracked files: README.md
$ git add README.md
$ git status   # Changes to be committed: new file: README.md

B4. Make your first commit with a real message, then check both git status (clean?) and git log (is your snapshot there?).

hint

The -m adverb from the grammar panel.

solution
$ git commit -m "Add README"
$ git status   # nothing to commit, working tree clean
$ git log

B5. Change README.md (append a line), then run the whole loop again: status → add → commit. Two commits in your log now?

hint

>> appends. The status this time says modified, not untracked.

solution
$ echo "Day two." >> README.md
$ git status   # modified: README.md
$ git add README.md && git commit -m "Add day-two note"
$ git log

I1. Create two files (script.py, notes.txt), stage both with one command, and commit them together.

hint

git add . stages everything new/changed in the current directory.

solution
$ touch script.py notes.txt
$ git add .
$ git commit -m "Add script and notes"

I2. Teach git to ignore clutter: make a .gitignore that ignores *.log files, create debug.log, and confirm git status doesn't mention it. Commit the .gitignore.

hint

.gitignore is just a text file of patterns, one per line — and it gets committed like any other file.

solution
$ echo "*.log" > .gitignore
$ touch debug.log
$ git status   # .gitignore untracked; debug.log invisible
$ git add .gitignore && git commit -m "Ignore log files"

I3. Publish: create an empty git-practice repo on GitHub (no README!), wire it as origin, and push. Refresh the GitHub page — your files are live.

hint

Workshop 1's Guide 3, Step 5 — remote add, then push with -u the first time.

solution
$ git remote add origin git@github.com:USERNAME/git-practice.git
$ git push -u origin main

I4. Close the loop across machines: clone the repo on your laptop, append a line to README there, push — then git pull on Fiji and watch the change arrive.

hint

Your laptop needs its own key on GitHub (Guide 2). Then clone → edit → add/commit/push → pull on the other side.

solution
# laptop:
$ git clone git@github.com:USERNAME/git-practice.git && cd git-practice
$ echo "From my laptop" >> README.md
$ git add README.md && git commit -m "Edit from laptop" && git push
# fiji:
$ git pull

A1. Branch like you mean it: create a branch figure-1, commit a change on it, switch back to main (your change vanishes!), then merge it in (it's back).

hint

git switch -c creates-and-switches. Merging happens from the branch you're standing on.

solution
$ git switch -c figure-1
$ echo "plot code" > figure1.py && git add . && git commit -m "Start figure 1"
$ git switch main   # figure1.py gone — different timeline
$ git merge figure-1   # timelines joined; it's back

A2. Practice the two undos everyone needs: (1) wreck README.md on purpose and restore it to the last commit; (2) stage a file, then change your mind and unstage it — without losing the edit.

hint

git restore for the working copy; git restore --staged for the loading dock.

solution
$ echo "OOPS" > README.md
$ git restore README.md      # back to last snapshot
$ echo "keep me" >> README.md && git add README.md
$ git restore --staged README.md   # off the dock, edit intact

A3. Read history like a pro: show the compact one-line log, then inspect exactly what changed in your latest commit.

hint

--oneline is a log adverb; git show narrates one commit.

solution
$ git log --oneline
$ git show HEAD   # the newest commit: message + exact changes
